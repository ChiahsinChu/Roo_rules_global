This document defines how to manage the development environment.

## 1. Package Manager Strategy

**Priority Chain:** `micromamba` > `mamba` > `conda` > `pip`

1.  **Binary Managers (Preferred):**
    - **Check Availability:** Use `micromamba` if available. If not, try `mamba`. If not, use `conda`.
    - **Execution:** Attempt to install the package using the available tool with `-c conda-forge`.
2.  **The "Last Resort" (Pip):**
    - **Trigger:** Use `pip` **ONLY** if:
      - The package is not found in `conda-forge`.
      - OR all conda-based tools fail/are missing.
    - **Safety:** Ensure the correct environment is active before running `pip install`.

## 2. Common Workflows

- **Create:** `<tool> create -n <name> python=3.10 -c conda-forge`
- **Install:** `<tool> install <pkg> -c conda-forge` || `pip install <pkg>`
- **Export:** `<tool> env export --no-builds > environment.yml`

## 3. Shell & Scripts

- **Strict Mode:** All shell scripts must start with `set -e`.

## 3. Build Systems

- **Python:** Use `pip install -e .` for editable installs during development.
- **C++:**
  - Build directory: `build/`
  - Command: `cmake -B build -S . && cmake --build build`
