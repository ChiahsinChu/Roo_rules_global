These rules govern the visual presentation, file structure, and organization of code generated by the agent.

## 1. General Formatting

- **Indentation:** Use 4 spaces consistently (unless project `.editorconfig` overrides). Do NOT use tabs.
- **Line Length:** Soft limit of 80 characters, hard limit of 100 characters (132 for Fortran).
- **Whitespace:**
  - Add a blank line between functions/methods.
  - Use spaces around operators (e.g., `a + b`, not `a+b`).
- **Trailing Commas:** Use trailing commas in multi-line arrays/objects where supported.

## 2. File Organization

- **Imports:**
  1. Standard/Built-in libraries.
  2. Third-party libraries.
  3. Internal project imports.
  - _Sort imports alphabetically within each group._
- **Order:** Imports -> Constants/Types -> Helper Functions -> Main Exports -> Tests.

## 3. Naming & Case Consistency

- **Filenames:** `snake_case` (Python/Fortran/C++) or `kebab-case` (JS/TS/General).
- **Variables:** `snake_case` (preferred for Python/C++/Fortran) or `camelCase` (JS/TS).
- **Constants:** `UPPER_SNAKE_CASE`.

---

## 4. Language Specific Rules

### Python

- **Style Guide:** Follow PEP 8 standards.
- **Docstrings:** STRICTLY use **NumPy-style** docstrings.

  - **Format:** Summary, Parameters, Returns, Raises.
  - **Example:**

    ```python
    def func(x):
        """
        One line summary.

        Parameters
        ----------
        x : int
            Description of x.
        """
    ```

- **Quotes:** Double quotes `"""` for docstrings, single quotes `'` for strings.

### C++

- **Style Guide:** Follow **Google C++ Style** generally.
- **Headers:** Use `#pragma once`.
- **Memory:** Use smart pointers (`std::unique_ptr`) instead of raw pointers.
- **Formatting:** Open braces on the _same line_.
- **Comments:** Use Doxygen style (`///` or `/** */`).

### Fortran (Modern)

- **Standard:** Free Form (`.f90`+).
- **Safety:** `IMPLICIT NONE` is **MANDATORY**. Use `INTENT` for all arguments.
- **Precision:** Use `ISO_FORTRAN_ENV` (`real64`); avoid `DOUBLE PRECISION`.
- **Syntax:** Keywords in `UPPERCASE` (e.g., `DO`, `IF`); variables in `snake_case`.
- **Operators:** Use symbols (`==`, `<=`) instead of `.EQ.`, `.LE.`.
- **Structure:** Place procedures inside `MODULE`s; avoid standalone subroutines.

### JavaScript / TypeScript

- **Quotes:** Single quotes `'` preferred.
- **Semi-colons:** Always use semicolons `;`.
- **Components:** Use `PascalCase` for React components.
